## Gameplay

- **Multi-recipe menu.** Let players serve more than one drink per day. Currently deferred — `Serving Today` plus a single hot/iced pair is intentionally simple. Revisit if testers feel constrained after the current UI lands.
- ~~**Per-recipe cup price.**~~ ✅ Done — `state.cupPrices` now holds a separate hot/iced price; switching hot ↔ iced remembers the price you charged last time.
- **Customer variety.** Right now `spawnCustomer` just rolls preferences from uniform ranges. Could introduce archetypes (sweet-tooth, espresso purist, ice fiend) that show up in different proportions with weather.
- **Day-over-day events.** Random one-off events at the start of a day — supplier sale, festival nearby, equipment breakdown — that tweak prices, demand, or capacity.
- **Spoilage / meltage.** Perishables degrade between days when it's warm out — there's currently no downside to over-buying since stock persists freely. Milk **spoils** above some temperature threshold; ice **melts** above a separate (probably lower) threshold. Tie the threshold to `state.weather.tempC` / the forecast so hot-day stocking becomes a real decision. The **Refrigerator** upgrade (see below) negates it. Lost stock should be surfaced clearly (e.g. on the report card) so it doesn't feel like a silent bug.
- **Allow keeping spoiled milk?** — Do we want to offer the choice to keep spoiled milk? It would either contaminate the entire supply or we keep track of how much is spoiled. How would spoiled milk affect customers? Can you mask it by adding more sugar?
- **Pastries with same-day staleness.** A second product line alongside coffee. Each morning the player stocks pastries — croissants, muffins, cookies — that customers can add to their drink order or buy on their own. Pastries go stale around midday (or after some game-clock threshold); stale stock either gets thrown out (lost-stock line on the report card alongside spoiled milk) or sold at a deep discount with a small hype penalty. Adds real bake-to-demand pressure to the buy phase: too few and you miss upsells; too many and you eat the loss when the clock turns. Per-pastry **ingredient choice** brings another lever — e.g. **butter vs margarine**, where butter is pricier (its own market band like the coffee/sugar/milk lines), customers prefer it (higher willingness-to-pay + better hype delta on taste), and margarine is cheaper but loses on both. Same recipe, different cost/quality call that scales with the prevailing fat market. The existing **Pastry case** upgrade below becomes the unlock that grants access to this whole system; deeper upgrades hang off it — a **bread-warmer / display case** that pushes the staleness threshold from midday to close-of-day (or eliminates it entirely), and synergy with **Refrigerator** for ingredient storage between days.
- ~~**Bulk discount on ingredient buys.**~~ ✅ Done — `economy.ts` now exposes `BULK_TIERS` (the single source of truth for the buy quantities + their discounts) and `bulkCost`; `attemptTrade` tiers the per-unit cost: Buy 5 at 100%, Buy 10 at 95%, Buy 20 at 90%. The visibly-cheaper totals on the larger buy buttons speak for themselves (no "save X%" tag — it was tried and removed as clutter). No partial buys: the buttons disable when you can't afford the full bundle. Still lives in tension with **Spoilage / meltage** above (bulk + perishables = risk) and pairs with **Refrigerator** (bulk + freezer = strategy).
- ~~**Hype rework.**~~ ✅ Mostly done — meter swings both ways, decays toward neutral, pulses on change, reputation bands and customer remarks. Open ideas: a streak counter for consecutive happy customers, or a "buzz" indicator (e.g. meter shake at high hype). Might be over-engineered given how picky customers already are.
- ~~**Out-of-stock warning mid-day.**~~ ✅ Done — cups-left HUD counter turns amber when low (≤3) and red when sold out, with a warning bubble over the shop and a one-shot chime.
- **Upgrade system.** Persistent purchases unlocked once you've banked enough cash, applied between days from the buy phase. Each one nudges a specific lever in customer/spawn/economy logic. Some seed ideas:
  - **Mascot.** Costumed character that patrols the sidewalk and physically drags reluctant pedestrians into the queue. Boosts `willStop` rate, especially on bad-weather days.
  - **Espresso machine v2.** Reduces coffee dose per cup (better margins) or speeds up the considering phase so the queue moves faster.
  - **Loyalty card.** Returning customers (same sprite seed) get a small price tolerance bonus — rewards consistency across days.
  - **Awning extension.** Customers in the queue don't flee in rain; smaller demand penalty when it's bad weather.
  - **Refrigerator.** Prevents spoilage/meltage (see **Spoilage / meltage** above) so milk and ice survive hot days. Lets the player safely stock ahead when prices are low regardless of the forecast.
  - **Weather forecast.** Reveals a multi-day forecast (next few days' conditions + temps) so the player can plan recipe choice and ingredient stocking ahead instead of reacting day-to-day. Backed by a **locked, stored forecast buffer** — evolve today's single `tomorrowWeather` into a rolling `forecast: Weather[]` that's sampled once, committed to `state` (and `save.ts`), and shifted forward at each day rollover — so what's shown is what actually happens. Could unlock progressively: today only → tomorrow → 3 days. Surfaces in the expandable weather chip. Strong synergy with **Refrigerator** (confidently stock ahead for a coming hot/cold spell) and the price sparklines (buy an ingredient cheap now for a recipe the forecast says you'll run later).
  - **Weather insights / analytics.** By default the weather chip shows loose qualitative "vibe" hints (busy vs quiet streets, the crowd leaning hot/iced, a cozy-rainy-day mood) — deliberately *not* hard numbers, so the player still discovers the price ceiling and exact demand by feel, which is the heart of the pricing loop. This upgrade reveals the underlying numbers from `weatherEffects` — foot-traffic %, the hot/iced bias, and the per-cup price tolerance — for players who want to min-max. **Bake-in:** the chip renders from a detail *tier* (`'vibe' | 'precise'`), and each insight is computed with both a vibe string and a precise string up front (see `chips/weatherChip.ts`); today the tier is hardcoded to `'vibe'`, and the upgrade just flips it (read from upgrade state). Pairs with **Weather forecast** as a "weather intel" upgrade track.
  - **Pastry case.** Unlocks the **Pastries with same-day staleness** system above — a side menu of croissants / muffins / cookies that customers add to their order or buy on their own, with the staleness clock, butter-vs-margarine ingredient choice, and bake-to-demand pressure that come with it. The simpler "pure upsell, no staleness" v1 of this could ship first as a pre-staleness milestone, but the eventual feature is the full system.
  - **Pup cups.** Adds a dog-walker customer type. Their dog gets a free (or cheap) whipped-cream cup; owner is delighted regardless of recipe fit. Strong hype generator, modest revenue.
  - **Tip jar.** Each happy customer drops a few cents on top of the cup price. Scales with hype.
  - **Cosmetics.** Awning colors, sign fonts, shop themes (e.g. "speakeasy", "tiki", "winter"). Pure flavor, no mechanical effect — or a tiny passive hype trickle if the theme matches the weather.

## UI / UX

- **Tutorial / onboarding.** First-time play-through is confusing per testing. A few inline tooltips on day 1 ("this slider sets dose", "this chip says today's market price") would go a long way.
- ~~**Price history sparkline.**~~ ✅ Done — a 5-day sparkline (`state.priceHistory`) now sits next to each ingredient's price chip, normalized to the band and colored by the last day's direction.
- **Report card depth.** End-of-day card shows totals and top complaint. Could also show which customer types walked by without buying — useful signal for recipe tuning.
- **Day-view stats.** Surface revenue-so-far (and maybe a conversion %) during the day, and/or a small daily summary in the buy phase. Keep it glance-able — the HUD already shows Sold / Walk-bys, so this should add signal without making the UI visually busy.
- **Affordances / accident-resistance.** Ongoing — make controls obvious and hard to trigger by accident. Known issues: dose sliders can be nudged when trying to scroll the page on mobile (tried `touch-action: pan-y`; Firefox Android still grabs the gesture on any horizontal drift). Likely fix is replacing sliders with `−`/`+` steppers or making the slider a read-only meter beside dedicated buttons. Same lens applies anywhere a tap could be misread (drag thresholds, confirm-on-destructive, clearer "you bought X" feedback — partly addressed by the buy-flash).
- **Multi-slot save system.** Today `save.ts` is hardcoded to one `coffee-shop-save` key — Continue / Save / Restore all operate on the same slot, so starting a New Game wipes whatever you had (now explicit in the title-screen confirm). A 2–3 slot system would let players park a long-running run in slot 1 while experimenting in slot 2. Implementation sketch: parameterize the localStorage key by slot id (`coffee-shop-save:0..2`), turn `loadGame()` / `saveGame()` / `clearSave()` into slot-aware variants, and surface a slot picker on the title screen (Continue and New Game both show a 3-card picker with last-played day / cash / hype previews). Pause-menu Save defaults to "current slot"; Restore reloads it. Probably want a "Save As…" affordance for copying a run into another slot.
- ~~**Confirm on Reset.**~~ ✅ Done — Reset now opens a styled in-game confirm modal (`confirmModal` in `ui.ts`) instead of wiping the game silently; cancel/click-outside/Esc all dismiss it.

## Tech / polish

- **Vector art.** `render.ts` is already shaped to allow swapping `emojiSprite` for vector implementations without touching scene code. Worth doing once the gameplay is settled.
- **Audio variety.** Single coin / bell / grumble samples; could use 2–3 variants of each to reduce repetition.
- **PWA install for iPhone fullscreen.** The in-pause-menu fullscreen toggle and the landscape-rotate prompt both rely on `Element.requestFullscreen()`, which iPhone Safari has *never* shipped (iPadOS Safari 16.4+ has it; Android Chrome and desktop browsers all do). Apple's intended path for iPhone fullscreen is the PWA route — a `manifest.json` with `"display": "fullscreen"`, plus a couple of icons and `<link rel="manifest">` in `index.html`. Once installed via Safari's share menu → "Add to Home Screen", the game launches from the home screen with no browser chrome at all, persistent across sessions. Doesn't replace the existing Fullscreen API code (which keeps working on Android/desktop) — both paths can coexist, the right one fires depending on platform. Also gets us a real home-screen icon as a side benefit.
